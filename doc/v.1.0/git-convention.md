# 📢 [Frontend] Git & GitHub 협업 가이드

원활한 프론트엔드 개발 협업을 위해 아래 컨벤션을 **엄격하게** 준수해 주세요.

## 1. 브랜치 전략 (Branch Strategy)

우리는 **Git Flow**를 간소화하여 사용합니다.

- `main`: 배포용 최종 브랜치 (Direct Push 🚫)
- `dev`: 개발용 메인 브랜치. 모든 기능 브랜치는 여기서 파생되고 여기로 합쳐집니다.
- `feat/...`: 개별 기능/페이지 개발 브랜치.

### 🚩 브랜치 명명 규칙

`타입/기능명` 형식을 따르며, 뒤에 `-page` 같은 접미사는 생략하고 **핵심 단어만** 사용합니다.
(모두 소문자 사용, 띄어쓰기는 하이픈 `-`)

| 타입         | 설명                          | 예시                                       |
| :----------- | :---------------------------- | :----------------------------------------- |
| **feat**     | 새로운 기능 개발, 페이지 생성 | `feat/login`, `feat/mypage`, `feat/search` |
| **fix**      | 버그 수정                     | `fix/header-layout`, `fix/login-error`     |
| **refactor** | 코드 리팩토링                 | `refactor/auth-hook`                       |

## 2. 커밋 메시지 컨벤션 (Commit Convention)

**"어떤 작업을 했는지"** 명확히 알 수 있도록 아래 규칙을 지켜주세요.

### 📝 기본 형식

```text
태그: 제목 (한글 추천)

[본문 - 선택 사항]
```

### ✅ 작성 규칙 (Rules)

1.  \*\*태그(Type)\*\*는 **소문자**로 작성합니다.
2.  태그 뒤에는 \*\*콜론(:)과 공백( )\*\*을 둡니다.
3.  제목 끝에 마침표(.)를 찍지 않습니다.
4.  내용은 **명령문**으로 작성합니다. (\~함, \~추가, \~수정)

### 🏷️ 프론트엔드 필수 태그 (Tags)

| 태그         | 설명                                        | 예시                                              |
| :----------- | :------------------------------------------ | :------------------------------------------------ |
| **feat**     | 새로운 기능 추가, UI 구현                   | `feat: 로그인 페이지 마크업 및 스타일링`          |
| **fix**      | 버그 수정                                   | `fix: 네비게이션 바 링크 이동 오류 수정`          |
| **design**   | CSS 등 사용자 UI/디자인 변경 (로직 X)       | `design: 메인 배너 폰트 사이즈 조절`              |
| **refactor** | 기능 변경 없는 코드 구조 개선               | `refactor: 회원가입 유효성 검사 로직 훅으로 분리` |
| **style**    | 코드 포맷팅, 세미콜론 누락 등 (코드 변경 X) | `style: 프리티어 설정 적용`                       |
| **chore**    | 패키지 매니저, 빌드 설정 변경               | `chore: 패키지 의존성 버전 업데이트`              |
| **docs**     | 문서 수정 (README 등)                       | `docs: 리드미 파일 실행 방법 추가`                |

## 3\. 작업 워크플로우 (Workflow)

### STEP 1. 최신 코드 가져오기 (`dev` 기준)

```bash
git checkout dev
git pull origin dev
```

### STEP 2. 기능 브랜치 생성

`dev` 브랜치에서 새로운 기능 브랜치를 생성합니다.

```bash
# 예: feat/login 생성 및 이동
git checkout -b feat/login
```

### STEP 3. 작업 및 커밋 (Commit)

작업 후 변경 사항을 커밋합니다. (컨벤션 준수 필수)

```bash
git add .
git commit -m "feat: 로그인 입력 폼 컴포넌트 구현"
```

### STEP 4. 원격 저장소로 푸시 (Push)

```bash
git push origin feat/login
```

---

## 4\. Pull Request (PR) 가이드

1.  GitHub에서 **New Pull Request** 버튼 클릭.
2.  **Base**: `dev` ⬅️ **Compare**: `feat/내-브랜치` 설정.
3.  **PR 제목**: 작업 내용을 대표하는 제목 (커밋 메시지와 유사하게).
    - 예: `feat: 로그인 페이지 UI 및 기능 구현`
4.  **내용 작성**: 작업 요약 및 스크린샷(UI 변경 시 필수) 첨부.
5.  리뷰어(Reviewers) 지정 후 생성.
